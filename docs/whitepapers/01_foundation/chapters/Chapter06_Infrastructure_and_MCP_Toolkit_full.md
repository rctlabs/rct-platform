[oai_citation:1‡Chapter06_Infrastructure_and_MCP_Toolkit.pdf.pdf](sediment://file_00000000588871fa9ee4c2005de6d623)  Up to now, the whitepaper has covered:  - **Chapter 1** — The 8 AI problems & context window crisis.  - **Chapter 2** — Ecosystem architecture (3 domains, layered stack).  - **Chapter 3** — Vault-1068 & RCTDB as the memory backbone.  - **Chapter 4** — 9 tiers & 36 algorithms.  - **Chapter 5** — Experiments & verification as the proof layer. [oai_citation:2‡Chapter06_Infrastructure_and_MCP_Toolkit.pdf.pdf](sediment://file_00000000588871fa9ee4c2005de6d623)  From FDIA and JITNA perspectives:  - Infrastructure is where **A (Architect/Architecture)** and **A (Algorithm/Agent)** become **real processes**.  - JITNA roles **D, A, M, R** are implemented as running services:  - *D* — data and knowledge access,  - *A* — algorithmic workers, routers, and tools,  - *M* — memory services (Vault-1068, RCTDB),  - *R* — reflection via metrics, logs, and verification.  ---  ## 6.1 Goals of the Infrastructure Layer  The Infrastructure Layer is designed to satisfy four main goals.  [oai_citation:3‡Chapter06_Infrastructure_and_MCP_Toolkit.pdf.pdf](sediment://file_00000000588871fa9ee4c2005de6d623)  ### 6.1.1 Run by a Single Architect / Small Team  - The system must be deployable and maintainable by **one primary builder** in early phases.  - This constrains complexity and enforces **22-container realism** for Phase 1:  - fewer moving parts,  - clear boundaries,  - easier diagnosis and iteration.

<!-- Page 3 of 15 -->


### 6.1.2 Extensible to Larger Deployments (33+ Containers)  - The base design should extend naturally to:  - more tenants (users, orgs, projects),  - more tools and domains,  - more experiments and test infrastructure.  - Scaling should happen by **splitting roles into finer services**, not by redesigning the architecture.  ### 6.1.3 Expose a Clean MCP-Style Interface to LLMs  - Tools such as:  - retrieval,  - RCTDB queries,  - verification,  - metrics,  are exposed via an **MCP-style toolkit**:  - each tool has a clear schema and contract,  - LLM agents can call tools in a predictable, safe way.  ### 6.1.4 Tie Runtime to Memory and Proof  - All important flows must be:  - **observable** (logs, metrics, traces),  - **recorded in RCTDB** (for experiments and history),  - **reachable by the experiment suite** (Chapter 5) for regression and comparison.  In short:  > The Infrastructure Layer is designed so that **every action can be traced back to > memory (Vault-1068, RCTDB) and evaluated in the Proof Layer.**  ---  ## 6.2 The 22-Container Current Design (Phase 1)  The **22-container design** is the practical, minimal infrastructure for Phase 1:  [oai_citation:4‡Chapter06_Infrastructure_and_MCP_Toolkit.pdf.pdf](sediment://file_00000000588871fa9ee4c2005de6d623)  - running the 9-tier algorithm stack,  - integrating Vault-1068 and RCTDB,  - supporting RCTLabs + Floating Assistant as a public front door,  - and performing structured experiments.  Conceptually, the 22 containers can be grouped as follows  (*names are conceptual, not final implementation names*).

<!-- Page 4 of 15 -->


### 6.2.1 Gateway & Frontend Group (3–4 containers)  - **Web Frontend (RCTLabs)**  - Delivers the public-facing site:  - landing pages,  - docs,  - the Phase 1 Floating Assistant UI.  - Communicates with the Assistant API gateway for all AI interactions.  - **Assistant API Gateway**  - Normalizes requests:  - from the web frontend,  - from internal tools and scripts.  - Applies:  - authentication,  - rate limits,  - basic logging and correlation IDs.  - **Static Assets / CDN Container (optional)**  - Serves:  - static documentation,  - UX assets,  - public reference materials.  These containers are intentionally **thin**; most intelligence and state live deeper in the stack.  ---  ### 6.2.2 Orchestration & Routing Group (3–4 containers)  - **Orchestrator / Router**  - Implements much of **Tier 7** (Orchestration & Routing):  - interprets **JITNA packets**,  - decides which tiers/algorithms to invoke,  - chooses models and tools based on policy, cost, and risk.  - Uses:  - FDIA/JITNA metadata from the request,  - manifest knowledge from Vault-1068,  - runtime policies from the Config Service.  - **Workflow Engine**  - Manages multi-step flows for:  - experiments and test campaigns,  - product use cases (e.g., Floating Assistant flows),  - background tasks (nightly runs, indexing, bulk evaluations).  - **Policy & Config Service**

<!-- Page 5 of 15 -->


- Stores:  - routing rules and strategies,  - cost and latency limits,  - environment configuration (dev/stage/prod).  - Acts as a **single source of truth** for runtime decisions.  At runtime, this group expresses the **A (Architecture)** side of FDIA:  > It is where the chosen architecture is turned into concrete, routable behavior.  ---  ### 6.2.3 Model & Tool Workers (4–5 containers)  - **LLM Worker Pool(s)**  - One or more containers that:  - connect to external model APIs (multi-provider),  - handle retries, backoff, and rate limits,  - abstract model differences behind a unified interface.  - **Specialized Workers**  - Optimized for:  - code generation,  - structured reasoning,  - or specific domains (e.g., infra configs, UX flows).  - **Tool Bridge (MCP Adapter)**  - An MCP-style adapter that:  - exposes tools as RPC-like endpoints with schemas,  - normalizes external APIs into a consistent interface,  - enforces safety constraints (rate limits, input validation, policy checks).  In JITNA terms, this group implements much of **A (algorithm/agent)** and part of **D (online data)**.  ---  ### 6.2.4 Retrieval & Memory Group (3–4 containers)  - **Vault Access Service**  - Read-only interface to **Vault-1068**:  - typically backed by object storage (e.g., S3/Hetzner),  - respects manifest structure and FDIA/JITNA tagging,  - can provide:  - raw files,  - pre-computed embeddings,  - graph views.


<!-- Page 6 of 15 -->


- **GraphRAG / Vector Index Service**  - Holds:  - vector embeddings,  - graph-based representations,  - semantic indexes.  - Serves **Tier 2** retrieval requests:  - search by tags, FDIA/JITNA role, content similarity, graph neighborhood.  - **RCTDB API**  - Provides controlled access to:  - experiment records and runs,  - delta logs and Architect decisions,  - user/project memory profiles.  These containers form the runtime bridge into the **memory layer** (Chapter 3):  > Vault-1068 and RCTDB are accessed via API services that respect > manifest and schema, not by ad-hoc file reads.  ---  ### 6.2.5 Experiments & Verification Group (3–4 containers)  - **Experiment Runner Service**  - Executes:  - test suites,  - batch evaluation runs,  - campaigns (e.g., nightly 200-test smoke checks).  - Integrates with:  - orchestrator,  - LLM workers,  - RCTDB logging.  - **SignedAI Verifier Service**  - Implements:  - scoring logic,  - rule-based checks,  - LLM-based verification for outputs.  - Writes:  - judgments,  - detailed rationale,  - risk tags  back into RCTDB.  - **Metrics Aggregator for Tests**  - Summarizes:  - experiment outcomes,  - failure rates,


<!-- Page 7 of 15 -->


- performance trends over time.  Together these containers form the **operational side of the Proof Layer** (Chapter 5).  ---  ### 6.2.6 Observability & Support Group (3–4 containers)  - **Central Logging Service**  - Ingests logs from:  - gateway,  - orchestrator and workers,  - RCTDB,  - experiment runner,  - verifiers.  - Encourages **structured, JSON-like logs** with:  - correlation IDs,  - FDIA/JITNA tags,  - test or request IDs.  - **Metrics & Dashboard Service**  - Exposes dashboards for:  - core metrics (latency, error rate, throughput),  - SLOs and SLIs,  - resource usage (CPU, memory, cost proxies).  - **Tracing / Telemetry Collector**  - Captures:  - end-to-end traces for complex flows,  - spans across tiers and containers.  - **Admin / Maintenance Tools**  - Scripts and small services for:  - backup,  - schema migrations,  - admin CLI or task runners.  These containers connect **operations** with **proof and memory**:  > “What happened” in infra is observable, linked to specific experiments, > and explainable in terms of FDIA/JITNA and Architect decisions.  ---  ## 6.3 The 33-Container Future Vision  The **33-container design** is a **logical extension** of the 22-container base,


<!-- Page 8 of 15 -->


not a different architecture. It adds capacity and isolation while preserving core concepts.  [oai_citation:5‡Chapter06_Infrastructure_and_MCP_Toolkit.pdf.pdf](sediment://file_00000000588871fa9ee4c2005de6d623)  ### 6.3.1 More Granularity per Responsibility  - Large roles are split into:  - per-domain verifiers,  - dedicated retrieval services for different knowledge clusters,  - separate workers for heavy vs light tasks.  For example:  - **SignedAI Verifier → 3 containers**  - content verification,  - code/infra verification,  - policy & governance checks.  - **GraphRAG Service → 2–3 containers**  - offline indexing,  - online query,  - graph maintenance.  ### 6.3.2 More Tenant / Domain Isolation  As RCTLabs, ArtentAI, and SignedAI evolve:  - each may:  - have its own frontend and API gateway,  - maintain partially separated backends,  - still share core infrastructure:  - Vault-1068,  - RCTDB,  - core orchestrator and Proof Layer.  This improves:  - security isolation,  - blast-radius control,  - per-product scaling.  ### 6.3.3 Dedicated Simulation & Sandbox Containers  - **Simulation clusters** for:  - staging new algorithms,  - running experiments on candidate architectures,  - safe “sandbox” environments that mimic production but with separate data.

<!-- Page 9 of 15 -->


- These containers are heavily integrated with:  - Experiment Runner,  - Metrics Aggregator,  - RCTDB test schemas.  ### 6.3.4 Extended Toolchain & Partner Integrations  - Dedicated containers for:  - external APIs (analytics, billing, data vendors),  - partner-specific connectors,  - internal-only tooling.  The rule remains:  > Conceptual structure (FDIA/JITNA, 9 tiers, Vault-1068/RCTDB) stays the same.  > Containers simply become more specialized for **scale, isolation, and reliability**.  ---  ## 6.4 MCP-Style Server & Toolkit Model  At the heart of the runtime interface between **LLMs** and **infrastructure** is an **MCP-style toolkit**.  [oai_citation:6‡Chapter06_Infrastructure_and_MCP_Toolkit.pdf.pdf](sediment://file_00000000588871fa9ee4c2005de6d623)  - Each tool is implemented as a **server** (often one container) that exposes:  - an input/output schema,  - a clear contract,  - safety constraints.  - LLMs (including orchestrator agents) are given:  - a **tool registry**,  - precise instructions on *when* and *how* to use each tool.  ### 6.4.1 Tool Categories in RCT  Typical tool categories include:  1. **Knowledge Tools**  - `vault.search`  - search Vault-1068 by:  - FDIA/JITNA roles,  - tags,  - full-text or semantic query.  - `vault.get_doc`


<!-- Page 10 of 15 -->


- retrieve a specific file or section (with metadata).  - `graphrag.query`  - query graph-based views of vault content.  2. **Memory & Experiment Tools**  - `rctdb.get_experiment` — fetch experiment records and outcomes.  - `rctdb.get_run` — inspect a particular run.  - `rctdb.log_event` — write structured logs for specific flows.  - `rctdb.get_delta_history` — inspect change logs for projects or capabilities.  3. **Verification Tools (SignedAI)**  - `signedai.verify_text` — verification for textual outputs.  - `signedai.verify_code` — run tests or static checks on code/infra artifacts.  - `signedai.score` — assign risk or confidence scores per output.  4. **Ops & Observability Tools**  - `ops.get_metrics` — check latency, error rates for services or containers.  - `ops.get_trace` — fetch trace data for a given request or correlation ID.  5. **Utility Tools**  - `util.diff` — compute diffs between documents or configs.  - `util.plan` — propose low-level steps given a high-level plan.  All tools share **consistent, JSON-like schemas**, enabling LLMs to:  - choose tools based on capabilities and constraints,  - interpret responses in structured form,  - chain multiple calls for multi-step reasoning.  ### 6.4.2 FDIA/JITNA View of Tools  From FDIA/JITNA perspective:  - Tools that mainly **provide data** → **D & M**  - (vault search, experiment retrieval).  - Tools that **transform or test** → **Δ, A, R**  - (planners, verifiers, diff engines).  - Tools that **update memory** → **M & R**  - (logging, delta integration, decision records).  Embedding this mapping into tool metadata enables:


<!-- Page 11 of 15 -->


- more intelligent routing (Tier 7),  - easier debugging (“which tools implement which roles?”),  - better interpretability for humans and systems.  ---  ## 6.5 Observability, Logging, and Integration with the Proof Layer  RCT treats **observability** as part of the architecture, not a bolt-on.  [oai_citation:7‡Chapter06_Infrastructure_and_MCP_Toolkit.pdf.pdf](sediment://file_00000000588871fa9ee4c2005de6d623)  ### 6.5.1 Logging  Key principles:  - **Structured over ad-hoc**  - Logs are:  - JSON-structured where possible,  - tagged with FDIA/JITNA roles,  - associated with JITNA packet IDs or test IDs.  - **Correlated across tiers**  - A single request’s journey:  - from gateway,  - through 9 tiers,  - into Vault/RCTDB,  - through SignedAI,  can be traced via correlation IDs.  ### 6.5.2 Metrics  The metrics layer tracks:  - **Per-tier performance**  - latency,  - error rate,  - distribution of success vs failure.  - **Per-tool usage**  - number of calls,  - error and timeout types,  - cost estimation proxies.  - **Experiment-centric dashboards**  - success/failure per:  - test case,


<!-- Page 12 of 15 -->


- test suite,  - experiment campaign,  - trend lines over time.  ### 6.5.3 Tracing  Tracing is especially important for:  - complex experiments (multi-model, multi-tool chains),  - multi-stage user flows (e.g., Floating Assistant actions that trigger retrieval + generation + verification).  Traces provide:  - visual timelines of which tiers were engaged,  - where delays or errors occurred,  - links back to:  - vault docs (specs, codex),  - RCTDB records (runs, deltas).  ### 6.5.4 Proof Layer Integration  The **Proof Layer** (Chapter 5) is deeply wired into operations:  [oai_citation:8‡Chapter06_Infrastructure_and_MCP_Toolkit.pdf.pdf](sediment://file_00000000588871fa9ee4c2005de6d623)  - Experiment runners emit:  - structured logs,  - metrics,  - traces,  all tagged with test case IDs and FDIA/JITNA roles.  - SignedAI verification results:  - appear in observability dashboards,  - can trigger alerts  (e.g., “spike in verification failures for capability X”).  This design means:  > operational health, experimental results, and architectural understanding  > live in one connected space, not in separate silos.  ---  ## 6.6 Infra ↔ Vault-1068 ↔ RCTDB ↔ 36 Algorithms

We can now summarize the runtime relationships among the main components.  [oai_citation:9‡Chapter06_Infrastructure_and_MCP_Toolkit.pdf.pdf](sediment://file_00000000588871fa9ee4c2005de6d623)  ### 6.6.1 Algorithms → Tools → Vault-1068 / RCTDB  - Tiers **1–4, 6–8** use MCP tools to request:  - vault search and document retrieval,  - experiment data and delta history,  - codex entries and design decisions.  - Examples:  - Tier 2 retrieval uses `vault.search`, `graphrag.query`.  - Tiers 5 & 8 use `rctdb.get_experiment`, `rctdb.get_delta_history`.  ### 6.6.2 Algorithms → SignedAI & RCTDB  - Tiers **4 and 6** (generation and packaging) send candidate outputs to:  - **SignedAI** via MCP tools for verification,  - **RCTDB** for logging and future analysis.  - SignedAI returns:  - scores,  - risk tags,  - suggested actions (accept/revise/block).  ### 6.6.3 Infrastructure as Supervisor  - The **Orchestrator** uses:  - JITNA role metadata from requests,  - FDIA factors and constraints,  - manifest knowledge from Vault-1068,  to decide:  - which tiers to invoke,  - which workers/tools to use,  - when to escalate to stronger verification.  ### 6.6.4 Vault-1068 — Mostly Read-Only at Runtime  - Vault-1068 is updated mainly by:  - Tier 8 processes (learning & memory integration),  - human Architect edits and curation.  - Runtime flows:  - **consume** vault content for retrieval, guidance, style, and specs,  - rarely write directly to the vault (writes are mediated via explicit update processes).  ### 6.6.5 RCTDB — Write-Heavy During Operations

- RCTDB stores:  - experiment outcomes,  - request logs and deltas,  - project/user memory,  - meta-governance decisions (Tier 9).  - This makes RCTDB the primary:  - **time axis** of the ecosystem,  - **source of truth** for what has actually been tried and observed.  From a systems perspective, RCT forms a **tight loop**:  > Architecture → Tools → Memory → Proof → Revised Architecture → …  with infrastructure providing the **rails and engines** that keep the loop safe, observable, and governed.  ---  ## 6.7 Summary and Bridge to Chapter 7  In this chapter, we have:  [oai_citation:10‡Chapter06_Infrastructure_and_MCP_Toolkit.pdf.pdf](sediment://file_00000000588871fa9ee4c2005de6d623)  1.
