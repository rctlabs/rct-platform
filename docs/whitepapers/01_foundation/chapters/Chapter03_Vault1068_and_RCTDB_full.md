The layer must serve **both simultaneously**, so that:  - a human architect can read a document end-to-end, and  - an AI agent can locate, filter, and combine documents programmatically.  3. **Make Context Addressable, Not Just “Stuffed into a Window”**  AI agents should be able to ask questions like:  - “Show me all design decisions about container count.”  - “Find experiments that revised Algorithm X in Tier 4.”  - “Gather all reflection (R) documents for the vault evolution 915 → 1010 → 1068.”  This requires:  - a **manifest** describing every file,  - **JITNA roles** (`I/D/Δ/A/R/M`) to express how each file participates in intent,  - **FDIA-aware structure** to differentiate Data, Intent, Architect, Future.  4. **Capture Evolution Over Time**  RCT is a living system. This chapter walks through how the ecosystem arrived at **Vault-1068** and **RCTDB**  to satisfy these requirements.  ---  ## 3.2 Vault Evolution: 915 → 1010 → 1068  The current vault of **1,068 files** is the result of a multi-stage evolution, not a one-shot export.  ### 3.2.1 Stage 1 — The 915-File Vault (Exploratory Foundation)  The earliest large vault (~915 files) emerged from gathering:  - research notes,  - algorithm concept drafts,  - experiment logs,  - UX sketches and wireframes,  - platform and business roadmaps.  Focus at this stage:  - **Breadth over structure** — capture as much relevant material as possible.  - Avoid premature deletion: keep overlapping variants, early drafts, and competing ideas.  Characteristics:  - Redundant files and overlapping content were tolerated.  - Naming conventions and directory structures were inconsistent.  - FDIA and JITNA roles were *implicit* and not yet encoded in metadata.  Despite the messiness, the 915-file vault established a crucial fact:  > The *volume* and *diversity* of knowledge required for RCT  > is far beyond what can be squeezed into a single “golden doc”  > or any model’s raw context window.  > An **architected memory solution** is required.  ### 3.2.2 Stage 2 — The 1010-File Vault (Consolidated Structure)  The next major step was the **1010-file vault**:


<!-- Page 4 of 17 -->


- Files were:  - grouped into clearer topical clusters (algorithms, infra, UX, experiments, etc.),  - assigned more consistent naming schemes,  - pruned or merged where simple duplication was obvious.  This stage introduced:  - early **directory conventions** (e.g., `01_Algorithms/`, `02_Architecture/`, `03_Experiments/`),  - initial **manifest-style lists** (CSV/JSON enumerating filenames and roles),  - clearer separation between:  - raw vs synthesized docs,  - conceptual specs vs experiment outputs.  Limitations:  - FDIA and JITNA were still not integrated as **first-class metadata**.  - The mapping between whitepaper chapters and vault files was partial.  - RCTDB (as an explicit DB layer) was not yet fully separated from the file vault.  ### 3.2.3 Stage 3 — The 1068-File Vault (FDIA/JITNA-Aligned)  The current **Vault-1068** is an evolution of the 1010-file vault with three key advances:  1. **Manifest-Driven Structure**  - Every file has a corresponding entry in **`MANIFEST_1068.json`**.  - The manifest knows:  - where each file lives (`paths.dest_path`),  - what its purpose is (`title`, `vault_group`, `section`),  - how it participates in FDIA and JITNA (`fdia_role`, `jitna_role`),  - which other docs it is related to (`related_docs`, `graph`).  The manifest became the **“index of indices”** — the structured map of the vault.  2. **FDIA/JITNA Tagging**  - Each file has:  - `fdia_role`: which component(s) of FDIA it primarily serves (D, I, A, F).  - `jitna_role`: which JITNA dimensions it mainly contributes to (I, D, Δ, A, R, M).  This transforms the vault from a plain folder tree into an:  > **FDIA/JITNA-aware memory substrate**  > that can be navigated by *role* and *function*, not only by filename.  3. **Whitepaper- and Product-Aware Layout**

- Many files are now explicitly structured to:  - support specific whitepaper chapters (Ch.1–8),  - back RCTLabs / ArtentAI / SignedAI UX flows with real specs and data,  - feed Vault-aware tools (Windsurf, RCT-Fusion, GraphRAG Coach).  #### 3.2.4 Why 1068 = “Minimal Complete Set” (for Now)  The number **1068** is not arbitrary. It emerged from a deliberate balancing act:  - **Too few files** →  - important distinctions collapse into vague “summary” docs,  - experiments blur into single reports,  - UX & product details vanish,  - the system loses *traceability* and *proof*.  - **Too many unconstrained files** →  - memory becomes noisy,  - FDIA/JITNA roles overlap chaotically,  - indexes become harder to maintain,  - agents waste context tokens on near-duplicates.  The 1068-file design is a **minimal complete basis** in the current phase because:  1. **Conceptual coverage**  - Every major concept in the RCT ecosystem (FDIA, JITNA, Architect genomes, 36 algorithms, 9 tiers, 3 domains, container design, UX roadmaps, pricing, experiments) has at least one **primary file** and often a **supporting file**.  2. **Lifecycle coverage**  - For each major design decision, there is:  - an intent/theory file (I/A),  - one or more experiment / data files (D/Δ),  - reflection / synthesis (R),  - and often a long-term memory or roadmap entry (M/F).  3. **Operational manageability**  - 1068 files, while large, are:  - indexable by `MANIFEST_1068.json`,  - maintainable by a single architect with the help of tooling,  - small enough that each file can be revisited in targeted passes (e.g., per domain / per tier / per FDIA role).  In other words:  > 915 was **too raw**,  > 1010 was **more structured but not fully semantic**,  > 1068 is the first vault where **structure, semantics, and coverage**  > are aligned enough to act as the **canonical base** for RCT Phase 1.

Conceptually, RCTDB consists of several key logical collections: 3.5.1 Experiments & Tests Stores: ● definitions of test cases (scenario, input space, constraints),  ● run histories (time, model versions, algorithm tiers involved),  ● metrics (accuracy, qualitative scores, latency, cost),  ● environment parameters (which container, which infra config). Enables: ● JITNA-style queries:  “What is Δ between the current FDIA spec and the version used in experiments from three months ago?”  ● trace-based governance:  “Which Architect decision led to increasing container count from 22 to 33,  and what experiments supported that?”  3.5.3 Memory & Profiles Stores: ● user profiles & preferences (public vs confidential usage, risk appetite, domains),  ● project histories (what intents have been pursued, what outcomes obtained),  ● repeated patterns of interaction (which JITNA configurations recur). 3.6 Vault-1068 & RCTDB as Response to the Context Window Crisis In Chapter 1, the Context Window Crisis was described as a 3-layer problem: 1. Storage — too much to fit into any single window.  2. Addressing — no language to find the right parts.  3. Temporal — knowledge and decisions change over time.  Vault-1068, MANIFEST_1068, RCTDB, FDIA, and JITNA jointly respond: 1. Storage  ○ Vault-1068 holds structured, curated knowledge (files).  ○ RCTDB holds structured, curated experience (records).  2. Addressing  ○ FDIA roles classify knowledge by conceptual function (D/I/A/F).  ○ JITNA roles classify by operational role (I/D/Δ/A/R/M).  ○ The manifest provides a programmatic index.  3. Temporal  ○ RCTDB’s deltas and experiment_runs track evolution of design and performance.  ○ Vault documents tagged with Δ and R explain how and why things changed.  Thus: The context window becomes a viewport into a well-structured space, rather than the container for the entire space. 3.6.1 Link Back to Chapter 1 (Problems) ● Problem 1 & 2: Context Window & Lossy Retrieval  → Vault-1068 + MANIFEST_1068 give structured storage and semantics.
